Задание 1. Кастомный функциональный интерфейс и обработка коллекций
Контекст
Часто в бизнес-логике нужно применять разные стратегии валидации, преобразования или фильтрации данных.
Вместо того чтобы писать множество перегруженных методов, можно создать универсальный метод,
который принимает поведение в виде лямбды.

Задача
Создай функциональный интерфейс `Transformer<T, R>` с единственным абстрактным методом `R transform(T input)`.
Подсказка: он похож на стандартный `Function<T, R>`, но мы создадим свой, чтобы потренироваться.

Реализуй класс `ListProcessor` с методами:
```text
<T, R> List<R> map(List<T> list, Transformer<T, R>> transformer) — применяет преобразование к каждому элементу.

<T> List<T> filter(List<T> list, Predicate<T> predicate) — оставляет только элементы, удовлетворяющие условию. Здесь используй стандартный Predicate<T>.

<T> void forEach(List<T> list, Consumer<T> consumer) — выполняет действие для каждого элемента.
```
В методе main продемонстрируй работу:

Возьми список строк ["java", "lambda", "stream", "functional"].

Отфильтруй строки длиной больше 5 символов.

Преобразуй каждую оставшуюся строку в её длину (используя свой `Transformer`).

Выведи результат в консоль через `forEach`.

Вопросы для размышления
Почему для фильтрации я предложил использовать стандартный` Predicate<T>`, а не свой интерфейс?

Какое преимущество даёт использование функциональных интерфейсов по сравнению с наследованием или переопределением методов?
